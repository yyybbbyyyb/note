## 背景

+ 状态栏业务是一套由 CMS 配置驱动、端上动态渲染的通用顶部状态栏系统，覆盖首页、左导航首页、详情页、Launcher、少儿/长辈模式等多个场景，支持账号、搜索、VIP、语音、多模式、时间 Logo、USB、蓝牙、手柄、HDMI、少儿管控、Rich 展开态等多类入口。
+ 当前状态栏链路涉及 StatusBarApi、CmsStatusBar、StatusBarRequest、StateInterpreter、StatusBarLayout、RichStatusBarLayout、StatusBarBackupDataManager 等多个核心模块，同时包含 CMS 请求、状态同步、组件上屏、Rich 展开、兜底缓存、首帧优化、页面布局适配等复杂逻辑。
 
## 整体架构

核心链路可以概括成：
+ 业务页面

+ CmsStatusBar

+ GetItemResponse / ItemGroup / Item

+ StatusBarLayout / RichStatusBarLayout 上屏

对应核心类：
+ **StatusBarApi.java**
	+ 提供实例化状态栏（CmsStatusBar）的方法
+ **CmsStatusBar.java**
	+ 真发 CMS 请求、解析回包、兜底数据、ACTIVATE_TYPE 二次确认、拼装 MutableState
+ **AbsCmsStatusBar.java**
	+ 加网络请求层：EventBus 事件触发、Handler 去重、mEnableRequest 开关、网络限流
+ **StatusBar.java**
	+ 把算法落地到真实 UI 容器、ViewModel 回收池、生命周期、状态驱动 sync
+ **StateInterpreter.java**、
	+ 状态同步的纯算法（上屏/移除/绑定/释放），5 个抽象方法留给子类

+ CMS 请求类：StatusBarRequest.java
+ 兜底数据管理：StatusBarBackupDataManager.java
+ 动画相关逻辑：RichStatusBarAnimUtils.java



## 实现细节

### 页面接入状态栏

页面一般不直接 new CmsStatusBar，而是通过 StatusBarApi.buildFromArgs()。

首页普通模式接入
+ 在 HomeUIFragment.java 中，首页会
	+ findViewById(tv_status_bar)
	+ StatusBarApi.buildFromArgs(...)
	+ setOnRichStatusBarCallback(...)
	+ setEnableRichStatusBar(...)
	+ StatusBarApi.setCmsRequestPage(...)
	+ StatusBarApi.setPage(...)
	+ setStyle(...)
+ 这里有几个重点：
	+ buildFromArgs(..., false)：表示先不立即发请求，通常用于首帧优化。
	+ setEnableRichStatusBar(StatusBarUtils.isEnableRichStatusBar(), true)：首页普通模式支持 Rich 状态栏。
	+ setCmsRequestPage()：影响 CMS 请求参数里的 page。
	+ setPage()：影响本地上报、展示逻辑等共享数据。

左导航首页接入
+ 在 LeftNavHomeUIFragment.java 中，左导航首页使用：
	+ StatusBarApi.buildFromArgs(activity,mStatusBarRoot,  mArgs,BackupScene.LEFT_NAV_HOME,false,true,filter)
+ 几个差异点：
	+ 使用 BackupScene.LEFT_NAV_HOME，说明左导航首页有独立兜底场景。
	+ inLeftNavHomePage = true，请求 URL 会带 left_nav=1。
	+ 当前代码里注释说明：新首页暂不支持 Rich 状态栏，所以调用 setEnableRichStatusBar(false, true)。
	+ 还传入了 IStatusBarItemFilter，说明左导航首页可以本地过滤某些状态栏入口。

详情页接入
+ 在 DetailVideoFragment/DoubleRowDetailCoverPageFragment 中，详情页会额外创建：
	+ RichStatusBarLayoutCalibrator
	+ RichStatusBarMaskAnimator
+ 也就是说详情页不只是展示状态栏，还需要处理 Rich 状态栏展开后内容区域下移、遮罩显示、焦点恢复等问题

> HomeUIFragment、LeftNavHomeUIFragment、DetailVideoFragment、DoubleRowDetailCoverPageFragment，阅读以上具体类代码，体会状态栏接入与配置，这些类中主要涉及状态栏rich状态的切换、布局、获焦等逻辑

### CMS 请求链路

请求由 CmsStatusBar.java 发起。

核心方法是： onPageResume() -> sendRequest() -> sendRequestImpl(args) -> new StatusBarRequest(...) -> InterfaceTools.netWorkService().get(...)

请求时机
+ 请求时机主要在 AbsCmsStatusBar.java 中控制：
	+ 页面 onPageResume() 后注册 EventBus。
	+ mNeedRequest = true 时会请求。
	+ setEnableRequest(true) 时，如果页面已 resume，也会触发请求。
	+ 账号变化、VIP 信息变化、支付状态变化、网络恢复、手动刷新事件都会触发重新请求。
+ 对应事件包括：
	+ AccountChangedEvent
	+ VipInfoUpdateEvent
	+ OnPayStatusChangedEvent
	+ NetWorkChangeEvent
	+ StatusBarRefreshEvent
+ 其中网络变化有 5 秒频控，避免部分设备频繁网络变化导致重复请求。

请求参数
+ 在 StatusBarRequest.java 中，状态栏请求是 POST。
+ 请求名：REQUEST_STATUSBAR_ITEMS
+ 请求 URL 会根据场景拼接参数：
	+ rich=1：老 Rich 状态栏信息
	+ rich=2：新 Rich 状态栏信息
	+ left_nav=1：左导航首页
	+ sub_scene=scene_child_left_nav：少儿左导航
	+ child_center_style=1：新少儿管控中心样式
	+ launcher_ui=1：Launcher 场景
	+ sub_scene=SCENEDESKPAGE：双桌面场景
+ 另外 CmsStatusBar.sendRequestImpl() 会往参数里塞：mode = MultiModeManager.getInstance().getMode()，所以后台可以按标准 / 少儿 / 长辈等模式返回不同状态栏配置。

### 请求返回数据结构

CMS 返回数据主要对应这些类：
+ GetItemResponse.java
+ ItemGroup.java
+ Item.java

核心结构是：
+ GetItemResponse
	+ Result
	+ left_group
	+ right_group
	+ fix_focus_item_type
+ ItemGroup 里是一组 Item。
+ 每个 Item 代表一个状态栏入口，关键字段包括：
	+ button_type → mType：状态栏入口类型，对应 StatusBarItemType
	+ normal_info → mNormalInfo：普通态样式
	+ focused_info → mFocusedInfo：获焦态样式
	+ action → mAction：点击行为
	+ priority → mPriority：优先级
	+ layout → mLayout：边距等布局信息
	+ report_info / dtReportInfo：上报信息
	+ rich_info → mRichInfo：Rich 展开态数据
	+ show_mode → mShowMode：展示模式
	+ extra → mExtra：额外参数

### CMS数据转化UI

核心在 CmsStatusBar.java 的 processItemGroup()。逻辑大致是：
+ processItemGroup(leftGroup, rightGroup, fixFocusType)
+ clearAllAddItem()
+ updateRichStatusBarInfo(...)
+ MutableState state = changeState()
+ 遍历 leftGroup
+ addItem(state, item, negativeOrder)
+ 遍历 rightGroup
+ addItem(state, item, positiveOrder)
+ 删除旧配置
+ 设置 fixFocusType
+ state.apply()

这里有几个非常关键的设计点：
+ 左右两组 order 规则
	+ 左侧状态栏入口：order = i - leftGroup.size()。所以是负数，越小越靠左。
	+ 右侧状态栏入口：order = i。所以是 0 或正数，越大越靠右。
	+ 这就是 CMS 的 left_group / right_group 最终映射到 UI 左右布局的方式。
+ 不是所有 Item 都直接上屏
	+ CmsStatusBar 里有一个 ACTIVATE_TYPE 列表。
	+ 命中这个列表的组件只会先执行：state.activate(type)，不直接 attach()。
	+ 比如：\_AD、\_VOICE、\_MULTI_MODE、\_USB、\_BATTERY、\_BLUETOOTH、\_GAME_CONTROLLER、\_HDMI、\_PERSON_CENTER、\_VCOIN、\_LOGIN、\_SYSTEM_SETTING、\_PARENT_CENTER、\_SCENE_DESK
	+ 这些入口不直接上屏的原因是因为需要本地二次判断，例如：
		+ 当前是否登录
		+ 是否有 USB
		+ 是否有蓝牙设备
		+ 当前模式是否允许显示
		+ 会员状态是否满足
		+ 设备能力是否支持
	+ 所以最终是否 attach 到屏幕，由对应 ViewModel / Item 自己判断。
	+ 普通组件则直接：state.attach(type)


### 状态系统

这部分是状态栏架构最核心也最抽象的一层。
+ MutableState
	+ MutableState.java 是“可变状态”。
	+ 它提供这些操作：
		+ config()：写入组件配置
		+ activate()：让组件运行
		+ attach()：让组件上屏
		+ detach()：让组件下屏
		+ deactivate()：停止组件
		+ removeConfig()：删除组件配置
		+ apply()：提交状态
+ FinalState
	+ FinalState.java 是“不可变状态快照”。
	+ 它包含以下核心集合：
		+ mItemConfigs：组件配置
		+ mActiveItems：已激活 / 已运行组件
		+ mAttachRequestedItems：请求上屏的组件
		+ mAttachedItems：真正已上屏组件
		+ mFixFocusType：默认获焦的状态栏入口类型

+ StateInterpreter
	+ StateInterpreter.java 负责把新旧 FinalState 做 diff，然后同步到 UI。
	+ 它的同步顺序是：
		+ updateItemConfig
		+ activateNewItems
		+ attachNewItems
		+ detachOldItems
		+ deactivateOldItems

CMS 数据不会直接操作 View，而是先变成状态；状态变化再由 StateInterpreter 统一解释为 UI 组件的创建、绑定、上屏、下屏和释放。


### UI 上屏链路

最终 UI 上屏在 StatusBar.java 中完成。

关键方法：
+ onStateChanged()
+ sync(newState)
+ getStatusBarItem(type)
+ activateItem()
+ attachItem()

组件如何创建：
+ StatusBar.getStatusBarItem() 会通过：InnerViewTypeConvert.convert()把 StatusBarItemType 转成具体 viewType，再从 ModelRecycledPool 中获取对应 ViewModel。
+ 所以新增一个状态栏入口时，通常不只是加 CMS 类型，还要保证：
	+ StatusBarItemType 有类型
	+ InnerViewTypeConvert 能转成正确 ViewType
	+ 对应 ViewModel / Component 已注册
	+ 组件实现了 StatusBarItem
+ attachItem() 会：
	+ 把组件 rootView 加到 StatusBarLayout
	+ 绑定 HolderTrace
	+ 绑定子 ViewModel
	+ 处理焦点 hint
	+ 做状态栏 Trace 上报辅助

这就是状态栏入口真正显示在屏幕上的地方。


### Rich 状态栏机制

Rich 状态栏可以理解为状态栏的“展开态”。普通状态栏是顶部一排小入口；Rich 状态栏获焦后，会展开成一排更大的入口卡片 / 内容列表。

核心类：
+ RichStatusBarLayout.java
+ RichStatusBarLayoutCalibrator.java
+ DetailPageLayoutCalibrator.java

开启方式
+ 业务页面调用：mTvStatusBar.setEnableRichStatusBar(enable, contentWillOffset)
+ 其中：
	+ enable：是否开启 Rich 状态栏
	+ contentWillOffset：Rich 展开后内容区是否需要下移

Rich 数据来源
+ 在 CmsStatusBar.processItemGroup() 开始时，会调用：updateRichStatusBarInfo(leftGroup, rightGroup, fixFocusType)
+ 它会遍历左右组里的 Item，把含有 mRichInfo 的 item 加到 RichStatusBarInfo 中。
+ 所以 Rich 展开态不是另一个接口，而是状态栏接口里每个 Item 自带的 rich_info 聚合出来的。

展开 / 收起触发
+ RichStatusBarLayout.java 中：
	+ 普通状态栏获焦：showRichStatusBar()
	+ 焦点离开 Rich 容器：showNormalStatusBar()
+ 它内部有两套动画：
	+ 简易版动画：showRichStatusBarSimple() / showNormalStatusBarSimple()
	+ 复杂版动画：showRichStatusBarComplex() / showNormalStatusBarComplex()
+ 展开时会：
	+ 显示 rich_item_list
	+ 隐藏普通状态栏
	+ 设置默认选中位置
	+ 请求焦点
	+ 触发曝光重算
	+ 通知外部 Rich 可见性变化

Rich 可见性事件
+ StatusBar.java 里会 post：RichStatusBarVisibilityEvent
+ 部分状态栏组件会监听这个事件，比如激励广告入口在 Rich 展开 / 收起时可能需要更新自身状态。

### Rich 展开后的页面布局适配

Rich 状态栏展开后，页面内容不能被遮挡，所以需要布局校准。

通用校准
+ RichStatusBarLayoutCalibrator.java 用于根据 Rich 状态栏展开状态移动内容区。
+ 核心方法：alignTopInRichStatusBar(visible)
+ 如果 visible = true：smoothTranslateTo(getInitialViewTop(statusBar) + getRichStatusBarContentOffset())

详情页校准
+ DetailPageLayoutCalibrator.java 是详情页专用版本。
+ 它会处理详情页顶部状态栏和内容列表的整体偏移，还提供：
	+ alignTop()
	+ alignTopInRichStatusBar(boolean visible)
	+ smoothTranslateTo(int translation)

### 兜底数据机制

状态栏有非常重要的兜底机制，避免无网、请求失败、首帧延迟请求时顶部状态栏空白。

核心类：
+ StatusBarBackupDataManager.java
+ MMKVBackupStorage.java

兜底分三层
+ L1 内存缓存
+ L2 MMKV 持久化缓存
+ L3 assets 本地兜底 JSON

什么时候加载兜底
+ CmsStatusBar.java 的 initCommon() 中：
	+ 如果无网络：loadDefaultStatusBarData()
	+ 如果 enableRequest = false：loadDefaultStatusBarData()
+ 也就是说，首帧优化场景下，状态栏可能先展示兜底数据，等外部允许请求后再请求最新 CMS 数据。

本地兜底文件
+ StatusBarBackupDataManager.java 中定义：
	+ statusbar_config/default_statusbar.json
	+ statusbar_config/default_statusbar_for_scene_desk.json

MMKV 存储 key
+ MMKVBackupStorage.java 中：
	+ SCENE_{scene}.data_version
	+ SCENE_{scene}.raw_data
+ 也就是说，每个场景会保存：
	+ 数据版本
	+ 原始 CMS 回包 JSON


### 首帧优化

为什么很多地方传 enableRequest=false，如StatusBarApi.buildFromArgs(..., false)
含义是：状态栏创建后先不立即请求 CMS，而是先用兜底数据，等页面合适时机再允许请求。
对应逻辑:
+ CmsStatusBar.initCommon()
+ setEnableRequest(false)
+ loadDefaultStatusBarData()
+ mNeedRequest = true

后面外部再调用：setEnableRequest(true)就会触发真实请求。
这种设计是为了避免首页 / 详情页首帧阶段被状态栏网络请求、数据解析、组件构建拖慢。


### 总结

页面通过 **StatusBarApi** 创建状态栏，**CmsStatusBar** 请求 CMS 并把 left_group/right_group 转成状态系统，**StateInterpreter** 根据状态差异创建、绑定、上屏或移除具体 **StatusBarItem**，**Rich 状态栏**则在普通状态栏获焦时基于 rich_info 展开，并通过 Calibrator **适配页面布局**；同时通过内存、MMKV、assets **三层兜底**保证弱网和首帧场景下状态栏可用。
